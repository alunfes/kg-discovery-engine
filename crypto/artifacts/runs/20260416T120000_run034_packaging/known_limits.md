# Known Limits — Production-Shadow Stack (run034)

*Last updated: 2026-04-16*

This document records limitations that are **understood, accepted, and not
currently planned for remediation**.  They are not bugs.  Each limit has a
documented impact level and a mitigation in place.

---

## L1 — Synthetic Data Only (High Impact)

**Description**: All validation runs (022–033) used synthetic Hyperliquid
market data generated by `MockMarketConnector` with deterministic RNG seeds.
No end-to-end test with real Hyperliquid tick data has been completed against
the delivery layer (push engine + family collapse + archive lifecycle).

**Impact**: Precision, stale rate, and reviews/day numbers are calibrated
against synthetic signal distributions.  Real market data will produce
different tier distributions, event frequencies, and regime durations.
The push trigger thresholds (T1 score ≥ 0.74, T2 count ≥ 3) may need
recalibration after 3–5 days of real-data shadow.

**Mitigation in place**: Run 017 validated the core detection logic against
real Hyperliquid data (10 cards, all beta_reversion, attention precision 100%).
The delivery layer is structurally sound.  A 7-day real-data shadow run is
the next required step before production.

**Accepted because**: The synthetic validation covers 20 seeds × 10 windows
= 200 simulation days of delivery behaviour.  The structural properties
(T1/T2/S1/S2/S3 logic) are data-independent.

## L2 — Funding / OI Coverage Gap (High Impact)

**Description**: The Hyperliquid `fundingHistory` API returns 0 records
unless a 7-day lookback is explicitly specified.  OI data comes from a
single snapshot replicated across all time steps (no real-time OI streaming).
Without funding time-series, the `positioning_unwind` grammar family
(E2 chain) cannot fire.  Without OI accumulation, OI-triggered hypotheses
are silent.

**Impact**: In real-data mode, the only grammar family that reliably fires
is `beta_reversion` (E1 chain).  Family diversity — a key health metric —
cannot be validated until funding and OI data gaps are resolved.

**Mitigation in place**:
- Use 7-day lookback for `fundingHistory` endpoint (run_018 finding)
- OI WebSocket subscription is the correct fix; not yet implemented
- Family diversity check in the weekly degradation checklist will surface
  this if real data shows < 2 families over a week

**Accepted because**: The delivery layer operates correctly on whatever
cards the detection layer produces.  The funding/OI gap is a detection-layer
issue, not a delivery-layer issue.

## L3 — positioning_unwind × HYPE High-Cadence Pair (Medium Impact)

**Description**: In all 20 soak-test windows (run_026), the
`positioning_unwind × HYPE` pair appeared 100% of windows, generating
~10 cards/window due to multi-asset hypothesis branching.  This is the
single highest-frequency pair in the synthetic distribution.

**Impact**: Without pair-level suppression, this pair will generate ~120
cards/day on synthetic data — all correctly attention-worthy (precision=1.0)
but cognitively repetitive for the operator.  Family collapse reduces this
from 10 to 1 digest entry/window (76% reduction), but the digest still
appears every window.

**Mitigation in place**: Family collapse currently aggregates all
`positioning_unwind × HYPE` variants into a single daily digest entry.
This reduces per-window burden from 10 cards to 1 collapsed entry.

**Remediation (not yet implemented)**: Add a pair-level suppression rule:
"suppress subsequent occurrences of the same (branch, family, asset) tuple
after the first occurrence per day."  This would reduce
`positioning_unwind × HYPE` from 20/day to 1/day.  Requires a new
suppression rule (S4) not yet in the engine.

**Accepted because**: With collapse active, the operator sees at most 1
entry per window for this pair.  The cognitive cost is manageable pending S4
implementation.

## L4 — Contradiction / expire_faster Rules Untested in Delivery Context (Medium Impact)

**Description**: The fusion rules `contradict` and `expire_faster` are
implemented and unit-tested (run_019) but have never fired in a longitudinal
or soak-test run.  All 20 soak-test windows produced 0 contradictions
(run_026).  This is because the synthetic event stream only generates
supporting events for the active hypothesis family — no opposing events
(e.g., `buy_burst` against a `beta_reversion` card) are present.

**Impact**: If real data produces opposing events and the `contradict` rule
fires, the operator will see a new event type not previously encountered.
The delivery notification format for contradictions has not been validated
for clarity.

**Mitigation in place**: Contradiction format is specified in
`crypto/artifacts/runs/run_019_fusion/example_contradictions.md`.  The
daily summary template includes a contradictions section.

**Remediation (not yet implemented)**: Inject a synthetic opposing-event
scenario in a future longitudinal run to exercise `contradict` and
`expire_faster` in the delivery context before real-data deployment.

**Accepted because**: The rule logic is correct and unit-tested.  The risk
is UX surprise, not missed signals.

## L5 — Tier Spread Absent in Synthetic Validation (Low Impact)

**Description**: In all longitudinal runs, 100% of cards end in
`actionable_watch` tier.  This is a synthetic data artifact: the mock event
stream consistently supports the active hypothesis, driving all cards to the
top tier.  Real market data will produce tier diversity (some windows with
monitor_borderline cards, some with research_priority only).

**Impact**: The monitoring guardrails for `reviews_per_day` are calibrated
against an all-actionable_watch synthetic deck.  A real deck with mixed
tiers may behave differently in terms of push trigger rate (T1 fires less
if fewer high-tier cards arrive).

**Mitigation in place**: The daily summary template tracks tier distribution.
Operators are instructed to expect tier spread in real data (playbook Section 2.2).

**Accepted because**: The structural properties of the trigger engine are
tier-independent for T2 and fallback.  T1 will fire less frequently in mixed
tiers, which is correct — it means the fallback will cover more reviews,
at the validated 45min cadence.

## L6 — Half-Life / Review Cadence Mismatch (Low Impact)

**Description**: Default half-lives (40–60 min) are shorter than the
recommended 120-min review cadence from run_026.  This means all cards
expire within a single window if the operator reviews every 120 minutes —
the structural stale count is 10/window (all prior cards expire).

**Impact**: The "stale card" count in the daily summary will always equal
the full prior-window deck size.  This is structural, not a decay failure.
It may look alarming to a new operator.

**Mitigation in place**: The playbook explicitly notes this is expected
behaviour (Section 4, "Stale card accumulation is structural").  The daily
summary template includes a fallback stale rate check at the actual review
cadence (45min fallback), not the 120min cadence.

**Remediation (optional)**: Extend half-lives to > 60 min across all tiers,
or shorten review cadence to ≤ 60 min.  Run_022 recommended either fix.
Not yet implemented because the 45min fallback effectively enforces a
45-min review cadence, making the HL/cadence mismatch less severe in
practice than in theory.

**Accepted because**: With the poll_45min fallback active, the effective
review cadence is 45 min (not 120 min), and cards with HL=40 do expire
before that cadence — but only marginally, and T1/T2 will have fired
before expiry in active markets.
