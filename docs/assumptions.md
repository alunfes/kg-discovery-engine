# 前提と制約

## 技術的前提

### KGデータモデル

1. ノードは一意のIDと人間可読なラベルを持つ
2. エッジは有向（source → relation → target）
3. ノードはドメインタグを持つ（cross-domain判定に使用）
4. 全データは合成（synthetic）トイデータセット — 実世界の正確性は保証しない

### 実験環境

1. 全実験は決定論的（random.seed固定、外部APIなし）
2. Python 3.11+、標準ライブラリのみ
3. シングルプロセス（並列処理なし）
4. メモリ制約なし（トイデータなので問題なし）

### オペレータ実装

1. align: 完全文字列マッチ + Jaccardトークン類似度のみ（埋め込みモデルなし）
2. compose: BFS深さ制限3ホップ（無限ループ防止）
3. evaluate: ルーブリック定義の単純ヒューリスティック（LLM評価なし）
4. analogy-transfer, belief-update: v0ではplaceholderのみ

---

## 実験的前提

### 評価の限界

1. 「有用な仮説」の定義はルーブリックに依存する — ルーブリック自体の妥当性は本プロジェクトでは検証しない
2. トイデータでの結果が実世界KGに一般化するかは未検証
3. サンプルサイズが小さいため統計的有意性は限定的

### 比較条件

1. 同一入力KGを全条件で使用（公平な比較）
2. 評価は全条件で同一ルーブリックを使用
3. C3（direct baseline）はプレースホルダー — 本フェーズでは参照値として扱う

---

## スコープ外（将来課題）

| 項目 | 理由 |
|------|------|
| WikiData / PubMed等の実KG | 規模・クリーニングコストが高い |
| LLM統合（GPT/Claude） | 決定論的動作が保証できない |
| 並列実験 | v0では不要 |
| WebUI | Streamlit等は外部依存になる |
| 統計的有意性検定 | サンプルサイズが小さすぎる |

---

## リスクと軽減策

| リスク | 影響 | 軽減策 |
|--------|------|--------|
| トイデータが単純すぎてH1-H4を検証できない | 高 | データを段階的に複雑化 |
| ルーブリックのheuristicが仮説品質を正確に反映しない | 中 | 複数ルーブリックで感度分析 |
| compose が有意義でない仮説のみ生成する | 中 | フィルタリング閾値の調整 |
| align がノイズを混入させる | 低 | threshold調整、手動検証 |
