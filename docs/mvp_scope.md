# MVP Scope — Hyperliquid KG Discovery Engine

## What the MVP Is

The MVP is a complete, runnable, deterministic pipeline that:
1. Constructs four knowledge graphs from deterministic synthetic market state data
2. Applies the align → union → compose → difference → rank operator chain to those KGs
3. Produces a set of scored hypothesis cards in JSON format
4. Classifies each card by secrecy level
5. Writes a review memo summarizing the run

The MVP is designed to prove that the pipeline works end-to-end on trading-domain data and to generate the first inventory of hypothesis cards that can be reviewed for economic plausibility. It does not connect to real market data. It does not execute any trades. It does not have a user interface.

---

## What Is Included

### The Four KG Types

All four KG types are included in the MVP:

| KG type | Status | Data source |
|---------|--------|-------------|
| Microstructure KG | Included | Synthetic (per-asset state transitions) |
| Cross-Asset KG | Included | Synthetic (cross-asset state correspondences) |
| Execution KG | Included (reduced scope) | Synthetic (execution quality states) |
| Regime KG | Included | Synthetic (macro regime states) |

The Execution KG in the MVP is simplified: it contains three execution condition nodes and four edges, enough to exercise the `difference` operator and generate execution-aware hypothesis candidates, but not enough to produce production-quality execution insights.

### The Operator Pipeline

The full operator chain is implemented and exercised:

| Operator | Status | Notes |
|----------|--------|-------|
| `align` | Production-ready | Jaccard similarity + synonym bridging |
| `union` | Production-ready | Namespace prefix for unaligned nodes |
| `compose` | Production-ready | BFS with configurable filters |
| `difference` | Production-ready | Regime-unique subgraph extraction |
| `rank` | Production-ready | 5-dimension rubric |
| `analogy_transfer` | Placeholder only | Not exercised in MVP |
| `belief_update` | Placeholder only | Not exercised in MVP |

### Hypothesis Inventory

The MVP produces a durable inventory of hypothesis cards stored in `runs/run_NNN_YYYYMMDD/output_candidates.json`. Cards include all fields defined in `docs/hypothesis_card_schema.md`. Secrecy classification is performed automatically based on the scoring rubric, with researcher review recommended before promoting any card to `private_alpha`.

### Secrecy Classification

Automatic classification is applied based on the decision criteria in `docs/alpha_vs_shareable_knowledge.md`. The MVP pipeline assigns an initial `secrecy_level` to every card; this is a starting point for human review, not a final determination.

## What Is Excluded

### Web UI
No dashboard, no visualization, no interactive review interface. Output is JSON and markdown. Visualization is out of scope for the MVP.

### Live Trading
The MVP has no connection to any trading system, execution API, or position management system. Hypothesis cards describe potential edges; acting on them requires a separate system not in scope here.

### Real Market Data Connectors
The market-data API at `localhost:8081` is the planned data source for production, but the MVP does not include a connector to it. All data is generated by `src/kg/toy_data.py` using deterministic synthetic fixtures.

### Broad Exchange Support
Only Hyperliquid-listed perpetuals are in scope. No spot markets, no options, no other exchanges.

### Trades, Order Book, Open Interest Data
The MVP does not use and does not plan to accommodate:
- Individual trade-level data (aggressor side, fill size, fill price)
- Order book depth (bid/ask levels, queue depth, imbalance)
- Open interest time series
- Liquidation events and volumes
- Cross-exchange data

These data types would enable qualitatively different hypotheses (particularly in the Execution KG) but are not available from the current market-data API.

### Statistical Significance Testing
The MVP does not include formal backtesting or statistical testing of hypothesis cards. Cards are generated and scored, not validated. Validation is a post-MVP activity.

## Data Available vs. Data Needed

This is an honest accounting of the data situation.

### Available (from market-data API at localhost:8081)

| Data type | Symbols | Resolution | Approximate depth |
|-----------|---------|------------|-------------------|
| OHLCV candles | HYPE, BTC, ETH, SOL | 1h, 4h | HYPE: ~12 months; others: 3+ years on Hyperliquid |
| Funding rates | HYPE, BTC, ETH, SOL | 8h | Same as OHLCV |

### Available in MVP (synthetic only)

The MVP uses synthetic deterministic data in `src/kg/toy_data.py`. This data is designed to have plausible structure (state transitions that make economic sense) but is not derived from real market observations. It is a scaffold for exercising the pipeline, not a source of real insights.

### Not Available, Would Enable Better Hypotheses

| Data type | What it would enable |
|-----------|---------------------|
| Order book depth | Real Execution KG: spread estimation, queue dynamics |
| Individual trades | Aggressor-side flow analysis, fill quality measurement |
| Open interest | Positioning-aware regime classification |
| Liquidation events | Liquidation cascade state nodes |
| Cross-exchange prices | Basis and arbitrage-related state nodes |
| Social sentiment | Narrative-driven regime states |

The absence of order book and trade data is the most significant limitation. It means the Execution KG is built on proxies rather than direct observation. Hypotheses about execution quality generated in the MVP should be treated with extra skepticism and classified conservatively.

## The Synthetic Data Approach

The MVP uses deterministic synthetic data for two reasons:

**1. The pipeline must be validated before connecting to real data.**
Connecting a complex multi-stage pipeline to a live data API during initial development creates confounded debugging: it is impossible to distinguish pipeline bugs from data quality issues. Synthetic data with known structure allows each operator to be verified independently.

**2. Reproducibility is a first-class requirement.**
Every experiment must produce identical output given identical input and random seed. Synthetic data with a fixed seed guarantees this. Real market data, even historical data, introduces version and access variability that complicates exact reproduction.

### Synthetic Data Design Principles

The synthetic KGs in `src/kg/toy_data.py` are designed with the following properties:

- **Plausible structure:** Node labels and edge types match the trading-domain vocabulary. Edges encode relationships that make economic sense (e.g., `funding_extreme_positive --leads_to--> vol_compression` is a plausible hypothesis about carry trade dynamics).
- **Non-trivial composition:** The KGs are designed so that `compose` generates non-obvious transitive hypotheses, not just direct-edge paraphrases.
- **Cross-asset alignment opportunities:** Microstructure KG nodes for different assets are designed to have alignable counterparts, enabling the cross-asset hypothesis generation that is the primary MVP goal.
- **Regime variation:** The Regime KG is designed to have distinct-enough node structure that `difference` produces a non-empty regime-specific subgraph.

### What the Synthetic Data Does Not Do

- It does not claim to represent real market behavior. Any hypothesis generated from synthetic data must be validated on real historical data before being classified above `internal_watchlist`.
- It does not simulate time series dynamics. State transitions are encoded as edges, not as time-ordered sequences.
- It does not guarantee that the hypotheses it generates are economically valid. The pipeline may produce high-scoring cards from synthetic data that have no real-world basis.

All hypothesis cards generated from synthetic data carry `source_streams: ["synthetic"]` and are automatically capped at `internal_watchlist` secrecy level pending real-data validation.

## Expected Output

A successful MVP run should produce approximately the following:

| Metric | Expected value |
|--------|----------------|
| Total hypothesis candidates generated | 15–40 |
| Candidates surviving rank filter (score >= 0.35) | 8–20 |
| Cards classified `private_alpha` (pending real-data validation) | 1–3 |
| Cards classified `internal_watchlist` | 3–7 |
| Cards classified `shareable_structure` | 3–6 |
| Cards classified `discard` | 2–8 |
| Cross-asset hypotheses (HYPE + at least one other asset) | 3–8 |

These ranges are estimates based on the size of the synthetic KGs and the expected behavior of the operator pipeline. Actual output depends on the specific KG structures used and the operator configuration in `run_config.json`.

## Success Criteria

The MVP is considered successful if the following conditions are met:

**Structural success (required):**
- The pipeline runs end-to-end without errors from synthetic data ingestion to hypothesis card output
- At least one hypothesis card is generated from a cross-asset path (HYPE + BTC, HYPE + ETH, or HYPE + SOL)
- The `run_config.json` + `output_candidates.json` + `review_memo.md` artifacts are all produced

**Content success (required):**
- At least 1 card is classified `private_alpha` (after researcher review, may require override)
- At least 3 cards are classified `shareable_structure`
- At least 2 cards are classified `discard` (demonstrates the filter is working)

**Reproducibility success (required):**
- Running the pipeline twice with the same `random.seed` produces byte-identical `output_candidates.json`

**Quality success (aspirational):**
- At least 1 card receives a `novelty_score >= 0.75` and `traceability_score >= 0.70` simultaneously
- At least 1 cross-asset card can be articulated as a coherent economic hypothesis in the `review_memo.md`

If the content success criteria are not met (e.g., all cards cluster in `discard`), the KG structures or operator filter settings should be adjusted before declaring the MVP complete. Generating only tautological output is a pipeline quality failure, not a success.

## Known Limitations

**No real-data validation.** All MVP output is from synthetic data. No MVP hypothesis card can be acted upon or promoted to `private_alpha` final status without real-data backtesting.

**No order book or trade data.** The Execution KG is built on approximations. Execution-related hypotheses from the MVP are particularly unreliable.

**No open interest or liquidation data.** Regime classification is based on price, volume, and funding only. Important regime signals (crowded positioning, liquidation cascades) are invisible to the system.

**No statistical testing.** Hypothesis scores are heuristic, not statistically derived. A card with `actionability_score = 0.85` is a researcher judgment, not a p-value.

**Synthetic data artifacts.** The synthetic KG is designed by humans and may contain structural biases that cause the pipeline to systematically favor certain hypothesis types. This bias will not be apparent until real-data comparison is performed.

**Single-run inventory.** The MVP produces a single inventory from a single run configuration. The inventory does not yet represent a multi-run, multi-condition comparison of hypothesis quality.

## Extension Points

The MVP is designed so that the following extensions can be added without architectural changes:

**Real data connector:** Implement a `MarketDataClient` class that calls `localhost:8081`, parses OHLCV and funding data, runs the state extraction layer, and produces `KnowledgeGraph` objects in the same format as `toy_data.py`. Swap the data source in `run_config.json`.

**Additional assets:** Add new symbols to the Microstructure KG by extending the state extraction configuration. The operator pipeline does not hard-code asset names.

**Additional state types:** Extend the state extraction layer with new rule definitions (e.g., open interest-based states when OI data becomes available). No operator changes required.

**Real backtesting integration:** Implement a `BacktestConnector` that takes a hypothesis card and runs a simple event-study backtest on historical OHLCV data. Write results back to `validation_status` and `reproducibility_score`.

**Regime-aware operator configuration:** Allow `run_config.json` to specify different operator filter parameters for different regime conditions, enabling per-regime pipeline tuning.

**Order book states:** When order book data becomes available, add `bid_ask_spread_state` and `queue_imbalance_state` node types to the Execution KG. The existing KG model, align, union, compose, and difference operators require no changes.
